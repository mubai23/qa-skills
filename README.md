# QA Skills 安装与使用说明

## 安装方式

### 方式一：插件安装（推荐，跨项目复用，支持自动更新）

**首次安装（3 步）：**

```
/plugin marketplace add mubai23/qa-skills
/plugin install qa-skills@qa-skills
/reload-plugins
```

**日常更新（已安装成员，2 步）：**

```
/plugin marketplace update qa-skills
/plugin update qa-skills@qa-skills
/reload-plugins
```

### 方式二：手动安装（独立 Skill，短名称，无自动更新）

```bash
# 项目级（只对当前项目生效）
cp -r skills/* .claude/skills/

# 用户级（所有项目生效）
cp -r skills/* ~/.claude/skills/
```

> **插件 vs 独立的区别**：插件安装后 skill 以 `/qa-skills:prd-analysis` 形式调用（带命名空间前缀）；独立安装后以 `/prd-analysis` 短名称调用。前者适合团队统一管理分发，后者适合个人快速使用。

## 5 个 Skill 说明

### 1. `prd-analysis` — 需求分析

**作用**：五步分析法（功能边界→业务规则→数据状态→等价类划分→风险点），输出带编号的业务规则清单（BR-xxx），为后续用例设计打基础。

**触发方式**：
```
分析这个需求：[粘贴 PRD/用户故事内容]
产品说要做一个商品搜索功能，支持按名称、分类、价格范围筛选，帮我分析一下
```

**输出**：BR-001 ~ BR-xxx 业务规则清单 + 等价类划分 + 风险点（R-xxx）+ 疑问列表（Q-xxx）

---

### 2. `case-design` — 测试用例设计

**作用**：基于需求文档或需求分析结果生成测试用例。默认运用正向/反向/边界值/等价类/场景法等策略，P0 不超过总数 20%。支持按 references 加载功能/接口/性能/自动化标准，支持按 assets 中的模板组织列格式。

**触发方式**：
```
# 基于需求分析结果生成（质量最高）
基于上面的需求分析，生成测试用例

# 直接从需求生成
为这个登录功能生成测试用例：[描述功能]

# 指定测试类型
为支付接口生成接口测试用例：[粘贴接口文档]
为订单查询接口生成性能测试用例

# 指定模板
参考柠檬班模板，为这个功能生成测试用例
```

**输出**：结构化用例表 + 用例设计说明 + 自检清单

**支持的标准**（`references/`）：
- `functional-testcases-standard.md` — 功能测试
- `api-testcases-standard.md` — 接口测试
- `performance-testcases-standard.md` — 性能测试
- `automation-testcases-standard.md` — 自动化候选

**Excel 导出**：`scripts/export_to_excel.py <markdown_file> [output_dir]`

---

### 3. `case-review` — 用例评审

**作用**：从覆盖完整性、用例质量、优先级合理性、冗余四个维度评审，输出具体到每条用例的问题列表，给出"可直接执行/修改后执行/需重新设计"的明确结论。

**触发方式**：
```
评审这批测试用例：[粘贴用例表格]
帮我评审这份测试用例文件
重点检查这批用例的边界覆盖和预期结果描述是否规范
```

**输出**：四维度问题明细 + 评审结论

---

### 4. `api-test` — 接口测试

**作用**：覆盖正常场景、参数异常、认证权限、业务逻辑、并发幂等五个维度，内置 SQL 注入/XSS/越权/敏感信息泄露等安全检查，输出可直接用于 Postman 的用例表和 curl 示例。

**触发方式**：
```
为这个接口生成测试用例：[粘贴接口文档/Swagger]
POST /api/user/register，参数：username、password、phone，帮我设计接口测试用例
为支付接口设计测试用例，重点覆盖金额边界和幂等性场景
```

**输出**：接口信息头 + 五维度用例表 + curl 示例 + Postman 导入提示

---

### 5. `bug-report` — 缺陷报告

**作用**：内置严重程度判断标准和"好/差对比"示例，对概率复现、接口 bug、UI bug、数据 bug 四类特殊情况有专门处理逻辑。

**触发方式**：
```
帮我写一个缺陷报告：在购物车页面删除商品后，数量没有更新，刷新后才恢复正常
把这段测试记录整理成标准缺陷报告：[粘贴记录]
帮我写缺陷报告，调用 /api/order/create 接口时返回了 500 错误：[粘贴请求和响应]
```

**输出**：完整缺陷报告（含严重程度判定、好/差对比、特殊场景处理）

---

## 推荐工作流

> **调用方式**：插件安装用 `/qa-skills:prd-analysis`（带命名空间前缀），手动安装用 `/prd-analysis`（短名称）。下文统一用插件形式示例。

### 功能测试标准流程

```
1. 拿到需求后
   → /qa-skills:prd-analysis 做需求分析
   → 确认分析结果，和产品/开发对齐疑问点

2. 写用例
   → /qa-skills:case-design 生成用例初稿
   → 人工审核和调整

3. 评审前
   → /qa-skills:case-review 做自检
   → 修改问题后提交正式评审

4. 测试执行中发现 bug
   → /qa-skills:bug-report 生成规范报告
   → 提交到缺陷管理系统
```

### 接口测试流程

```
1. 拿到接口文档
   → /qa-skills:api-test 生成接口测试用例
   → 导出测试数据用于 Postman/自动化

2. 发现接口 bug
   → /qa-skills:bug-report 生成缺陷报告
```

### 没有代码权限时

以上 5 个 Skill 全部适用，不需要访问源代码：
- `prd-analysis`：输入需求文档
- `case-design`：基于需求分析或功能描述
- `case-review`：输入已有用例
- `api-test`：输入接口文档或描述
- `bug-report`：输入缺陷现象描述

## 提升效果的技巧

**给越具体的输入，输出质量越高**

```
# 一般效果
分析登录功能的需求

# 更好的效果
分析以下登录需求：
- 支持手机号 + 密码登录
- 密码 6-20 位，包含字母和数字
- 连续错误 5 次锁定账号 30 分钟
- 支持"记住我"功能，7 天内免登录
```

**先做需求分析再设计用例**

直接说"生成用例"会跳过需求理解，容易遗漏边界场景。分两步做，用例质量明显更高。

**用例评审和设计分开做**

不要在设计的同时就做评审，先生成完整的用例集，再统一评审，效率和质量都更好。

**保留需求分析结果**

把 `prd-analysis` 的输出保存下来，后续用例评审时可以用来对照，检查设计时是否遗漏了分析中识别的场景。
